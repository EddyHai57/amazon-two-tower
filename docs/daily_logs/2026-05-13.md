## 用户塔 mean pooling 轻量升级

### 本次目标

- 只做用户历史 item embedding mean pooling 的轻量 Two-Tower 升级。
- 不做 Transformer、attention pooling、SASRec、LogQ、负采样大实验、hard negative mining、hybrid retrieval、text-enhanced、item text embedding 或 Faiss 新实验。
- 不覆盖 ID-only baseline checkpoint 或 outputs。

### 新增文件

```text
scripts/train_mean_pool_two_tower.py
configs/two_tower_movies_tv_5core_mean_pool_1epoch.yaml
```

### 模型口径

- item tower：仍为 `item_idx` embedding。
- user tower：`user_id embedding + history_weight * mean(history item embeddings)`。
- `history_max_len=50`，按 `timestamp` 取每个用户最近的 train history item。
- 训练时 history 来自 train split，并对每个训练样本排除当前正样本 item，避免直接把 target item 放进用户历史均值。
- valid 时 history 来自 train split。
- 本轮不运行 full test；后续如运行 test，应使用 train + valid history。

## Hard Negative Mining smoke（Text + Mean Pooling τ=0.15）

### 重要声明

**这是 smoke test，不是最终模型。**
- 不替换当前主模型（Text + Mean Pooling τ=0.15，full test Recall@50=0.076337）。
- Hard Negative Mining 用于探索相似 item 之间的细粒度区分能力，不是用来解决 true new-user cold start。
- 只跑 1epoch limited valid eval，不做 full valid/test、Faiss、20epoch。

### 本次目标

- 验证在 Text + Mean Pooling τ=0.15 架构上加入 Hard Negative Mining 训练链路是否能跑通。
- 观察 1epoch limited eval 是否有继续实验价值。

### Hard Negative 构造方式

- 使用已有 `outputs/item_text_embeddings/movies_tv_5core/item_text_embedding.npy`。
- 训练开始前一次性预计算 HN table：对 153977 个 item，分块（chunk_size=512）在 GPU 上计算 text cosine similarity，每个 item 保存 top-5 最相似 item（排除自身）。
- HN table 构建耗时约 5 秒（GPU 加速）。
- HN table 大小：`(153977, 5)` int32 ≈ 3MB。
- 训练时每个 positive item 从其 top-5 candidates 中选取 2 个：
  - 排除 positive item 自身。
  - 排除该 user 在 train history 中已交互过的 item（降低 false negative 风险）。
- 99.9%（4315539/4319438）的训练样本成功匹配到至少 2 个有效 hard negatives；只有 0.1% 的样本因 history 重叠而跳过。

- 核心假设：top-50 nearest neighbors 可能太 hard（false negative 风险 + 初期梯度冲突），从 rank 50–200 的中等相似 item 中采样应更稳定。

### Semi-hard Negative 构造方式

- 加载当前最终模型：`outputs/text_mean_pool_tau015_20ep/checkpoints/best_model.pt`（epoch=20）。
- 导出所有 153977 个 item 的 L2 normalized item embeddings（dim=64）。
- 使用 **Faiss IndexFlatIP** 搜索每个 item 的 top-300+2 neighbors（inner product = cosine similarity for L2 normalized vectors）。
- 排除自身后，**跳过 rank 0–49（最相似的 50 个），取 rank 50–199 作为 semi-hard candidate pool**。
  - rank 0-indexed：rank 0 = 最相似的 neighbor（自身排除后）。
- HN table 形状：`(153977, 150)` int32；**100.0% 有效条目**（每个 item 均有 150 个 semi-hard candidates）。
- HN table 构建耗时：**93.20 秒**（比 top-50 model-based 68.91s 更长，因 search top-302 neighbors）。
- 训练时每个正样本从 semi-hard pool 中选取 2 个有效 candidates：
  - 排除 positive item 自身。
  - 排除该 user 在 train history 中已交互过的 item。
- **100%（4319438/4319438）的训练样本成功匹配到至少 2 个有效 semi-hard negatives**。

## Time-decay Mean Pooling smoke（user history length 诊断驱动）

### 重要声明

- 目的：验证 time-decay weighted mean pooling 是否比 simple mean pooling 在 epoch1 limited valid 上更强。

### 设计动机

User history length bucket diagnostic（来自 `scripts/eval_user_history_buckets.py`）显示：

| --- | ---: |
| 3–5 | 0.086826 |

长历史用户 Recall@50 反而更低。假说：simple mean pooling 将多兴趣历史平均成模糊向量，导致信号稀释。

### 本次改动

相比最终主模型（Text + Mean Pooling τ=0.15）：

- **仅改变 user tower 中的 history pooling 方式**。
- item tower 完全不变。
- 所有训练超参不变（temperature=0.15, batch_size=4096, lr=0.001, seed=42 等）。

| 模块 | 旧版 | 新版 |
| --- | --- | --- |
| history pooling | `hist_pool = masked_mean(item_embs)` | `hist_pool = masked_weighted_mean(item_embs, decay_weights)` |
| decay formula | — | `weight_k = 0.8^(seq_len-1-k)` |

### Time-decay 权重设计

- `decay_rate = 0.8`（固定，非可学习参数）
- History 序列按时间从旧到新排列，最新 item 在末尾（index=seq_len-1）
- `weight_k = decay_rate ^ (seq_len - 1 - k)`
  - 最新 item（k=seq_len-1）：weight = 0.8^0 = **1.0**
  - 前一个 item（k=seq_len-2）：weight = 0.8^1 = **0.8**
  - 最老的 item（k=0）：weight = 0.8^(seq_len-1) = **最小**
- `hist_pool = sum(weight_k × emb_k × mask_k) / sum(weight_k × mask_k)`
- 无历史用户：weight_sum clamp_min(1e-8)，hist_pool ≈ 零向量（与 simple mean pooling 兼容）

由于归一化，位置权重 `decay_rate^(seq_len-1-k)` 等价于 `decay_rate^(L-1-k)`（L 为有效历史长度），两者比值不变。

**无新可学习参数**（trainable params = 41715840，与 baseline 相同）。

## Time-decay User History Bucket Diagnostic

**日期：2026-05-14**

### 动机

full test Recall@50 整体提升 +2.59%，但未验证提升来源。此前 simple mean pooling bucket diagnostic 揭示长历史用户（>20）Recall@50 最低（0.042312），被识别为主要痛点。Time-decay pooling 的设计动机正是修复这一问题。本次 diagnostic 验证 time-decay 是否真的改善各桶，尤其 >20 桶。

### 客观结论

1. **所有三个 evaluated bucket 均提升**，time-decay 是更稳健的 user history aggregation。
2. **>20 bucket 提升幅度最大**（+4.66% relative），验证了设计动机：simple mean pooling 会稀释长多兴趣历史，time-decay 聚焦近期偏好对长历史用户帮助最显著。
3. 3-5 bucket 提升 +2.26%，说明 time-decay 对短历史用户也略有改善（而非牺牲短历史换取长历史）。
4. 各桶提升幅度（约 +0.002 绝对）与 overall full test 提升（+0.001978）基本一致，说明提升均匀分布于各 history 长度桶，不是集中在某一子群。
5. 没有重新训练。没有 Faiss。没有 HNM。没有 decay_rate sweep。
6. 这不是 true new-user cold-start 评估：0 和 1-2 bucket 在本 5-core 处理后的 split 中无 non-cold test users，cold-start 问题需要额外 onboarding 信号或内容 fallback 策略。

### 综合结论

Time-decay pooling 的设计动机得到验证。full test 整体提升来自各 history 桶的均匀改善，>20 长历史桶的相对提升最大。建议正式采用 Time-decay Text+MP τ=0.15 decay=0.8 作为新的候选最终主模型。
