## 结论

- valid-test gap 在所有 item popularity bucket 中都存在；长尾 bucket 的绝对 Recall@50 最低，但 gap 不只集中在长尾。
- user history length 分桶中也普遍存在 gap；valid/test 的用户历史分布相同，因此 gap 不是由用户历史长度分布差异造成。
- ItemCF 相比 Two-Tower 的主要优势来自更高 popularity target 上的局部共现关系，`itemcf_hit_only` 的 target item popularity median 为 517。
- Two-Tower 仍有 ItemCF miss 但自己 hit 的样本，`two_tower_hit_only` 为 5936。
- rank sanity check 与已有 full eval 指标一致，未发现新的 evaluation bug 迹象。
- 当前更像是 ID-only 表达能力不足叠加 test target 更长尾、用户兴趣随时间漂移，而不是单纯训练轮数不足。
- 仍不建议直接启动 20/25/30 epoch 长训。

结论：

- ItemCF 在同一 clean split 上也存在明显 valid-test gap，幅度接近 Two-Tower 的 valid-test gap。
- 这说明当前大 gap 更可能来自 clean split 本身的 valid/test 难度差异或时间推进带来的目标变化，而不是 Two-Tower pipeline 独有 evaluation bug。
- Two-Tower test 仍明显低于 ItemCF test，因此 Two-Tower 模型能力问题仍存在，但 valid-test gap 本身不能只归因于 Two-Tower pipeline。
- 后续不建议继续把 Two-Tower eval bug 作为第一优先级。
- 后续不建议直接进入 20/25/30 epoch 长训。
- 下一阶段建议讨论 `text-enhanced item tower`、popularity correction、longer training ablation 的优先级。
- 当前更推荐先做 `text-enhanced item tower` 作为主线增强。
